# Journey of a Single Transaction: An End-to-End Walkthrough

To truly understand this Big Data pipeline, let us follow the exact lifecycle of a single synthetic transaction—from the millisecond it is created to the moment it is flagged as fraud on the dashboard.

---

## Stage 1: Synthetic Generation (The Python Producer)
*Where it starts: `project-copy-producer-1` container*

1. **How it's generated:** A separate Python process (`src/producer/main.py`) continuously loops, creating simulated credit card transactions. 
    - **Demographics:** It randomly pulls from a pool of 100 users (`user_id` 0-99) and assigns them to specific geographical coordinates and terminals.
    - **Fraud Injection:** To guarantee the ML model has something to catch, the script randomly injects "Impossible Travel" anomalies (e.g., triggering a transaction in New York, and 5 minutes later, one in London) or "Ring Fraud" patterns.
2. **Payload Size:** Each raw transaction is a relatively small JSON payload (roughly 300–400 bytes). However, because the producer generates them continuously (e.g., 5-10 per second), the streaming data volume quickly scales to megabytes per minute.
3. **Encryption:** Before leaving the container, the JSON payload is passed through the `cryptography.fernet` library. It uses **Symmetric AES Encryption**, utilizing the `ENCRYPTION_KEY` defined in `docker-compose.yml`. The payload becomes an unreadable byte string.

## Stage 2: Streaming Ingestion (Apache Kafka Cluster)
*Tech Stack: Apache Kafka, Apache ZooKeeper*
*Ports: Kafka Brokers (`9092`, `9093`, `9094`), ZooKeeper (`2181`)*

4. **The Hand-off:** The Producer's Kafka Client connects to the Kafka cluster on port `9092`. It publishes the encrypted byte string to a specific "topic" called `transactions`.
5. **Cluster Dynamics:** This Kafka cluster is highly available, consisting of 3 separate Broker containers (`kafka-1`, `kafka-2`, `kafka-3`). They are coordinated by an Apache ZooKeeper container running on port `2181`, which elects partition leaders and handles broker failures.
6. **Data Storage:** The encrypted transaction is temporarily persisted on Kafka's ephemeral disk volume. Even if the downstream processing engine crashes, Kafka holds onto the data.

## Stage 3: Decryption & Validation (Secure Gateway)
*Tech Stack: Python FastAPI/Kafka-Python, Redis*
*Ports: Redis (`6379`), Gateway API (`8000`)*

7. **Consumption:** The `secure-gateway` container actively subscribes to the Kafka `transactions` topic. It pulls the new encrypted transaction off the queue.
8. **Decryption:** It uses the same AES `ENCRYPTION_KEY` to decrypt the byte string back into readable JSON.
9. **Validation (Redis Check):** The gateway connects to the `redis-ram-cache` container on port `6379`. It checks the transaction's IP address against a known "Fraud Ring" list stored in Redis. If it matches, it flags `is_ring_fraud = 1` directly onto the JSON payload.
10. **Re-publishing:** The decrypted, semi-validated transaction is pushed *back* to Kafka, this time onto the `transactions_clear` topic for the main processing engine to handle.

## Stage 4: Micro-Batch Feature Engineering (Apache Spark)
*Tech Stack: PySpark (Structured Streaming), Spark Standard Cluster Manage, Pandas*
*Ports: Spark Master UI (`8080`), Master Node (`7077`), Workers (`8081`, `8082`)*

11. **The Engine:** The `spark-inference` container runs a continuous Spark streaming job. It connects to the `transactions_clear` Kafka topic.
12. **Cluster Topology:** Behind the scenes, the Spark job connects to the `spark-master` node (port `7077`), which distributes the workload across two `spark-worker` nodes to parallelize the heavy data lifting.
13. **Micro-Batching:** Spark doesn't process one transaction instantly; it waits ~2 seconds, groups multiple transactions together into a "Micro-Batch" (a Spark DataFrame), and processes them together to maximize throughput.
14. **Complex Transformations (The "Math"):** 
    - **Time Features:** Spark converts the transaction's `timestamp` into cyclical features (`hour_sin`, `hour_cos`) so the ML model understands that 11:59 PM is close to 12:01 AM.
    - **Velocity Metrics:** Using PySpark Window functions, it groups by `user_id` and calculates `tx_count_1h` by looking back at the last 60 minutes of data.
15. **Scaling:** The numerical `amount` column is passed through a Scikit-Learn `RobustScaler` (loaded from `/opt/spark/models/scaler.pkl`) to normalize massive monetary outliers.

## Stage 5: Machine Learning Inference (XGBoost)
*Tech Stack: XGBoost (Python)*

16. **The Prediction:** The Spark job loads the trained ML model from the shared volume (`/opt/spark/models/model.json`). It throws the engineered 14-feature row into the XGBoost classifier. 
17. **The Verdict:** The model utilizes its boosted decision trees to output a binary prediction: `0` (Safe) or `1` (Fraud). Let's assume this transaction was injected as an anomaly, so XGBoost predicts `1`.
18. **Heuristics:** Spark combines the ML verdict with hardcoded rules. It calculates the `distance_from_center` using the Haversine formula based on the GPS coordinates. It assigns a unified `is_fraud` flag and writes a human-readable `fraud_reason`.

## Stage 6: Fast Storage & Output (The RAM Disk)
*Tech Stack: Docker tmpfs (RAM Volumes), Unix File System*

19. **The Heartbeat Checkpoint:** To survive crashes, Spark writes its stream state (offsets, watermarks) to a shared Docker volume `spark_checkpoints_vol` mapped to `/tmp/spark-checkpoints`.
20. **Exporting for the UI:** Spark takes the final DataFrame of processed transactions and exports it. To ensure zero-latency reads for the web UI, it writes exactly two files to a `dashboard_ram` shared memory volume (literally writing to computer RAM, not hard drives):
    - `recent.json` (Containing the full transaction row for the table).
    - `live_features.json` (Containing *only* the specific 14 numerical features the model saw, used for SHAP explainability).

## Stage 7: Visualization & Explainability (The Web Dashboard)
*Tech Stack: Flask, HTML/JS/CSS, SHAP*
*Ports: Web API (`3000`)*

21. **The Display:** A user sitting on a browser views the frontend interface. The browser fires Javascript `fetch()` requests every 2 seconds to `http://localhost:3000/api/live`.
22. **The API Response:** The Flask web server reads the `recent.json` file from the RAM volume and instantly serves the payload. The transaction flashes red on your screen because `is_fraud = 1`.
23. **The "Why" (🧠 SHAP Click):** When the user clicks the "SHAP" button on that specific row, a request hits `/api/explain/<transaction_id>`.
24. **Reverse-Search & Tree Deconstruction:** 
    - The Flask backend opens the `live_features.json` RAM file and searches it in *reverse* (from newest to oldest) to instantly find the transaction's 14 exact features.
    - It passes those 14 isolated numbers into `shap.TreeExplainer(model)`.
    - SHAP mathematically deconstructs the XGBoost decision path and calculates exactly which features influenced the `1` prediction (e.g., "Terminal Fraud Rate was massive, contributing +3.5 to the final score").
    - The backend returns this matrix, and the frontend dynamically charts it on a bar graph popup.

## The Background Orchestration (MinIO, Airflow, & MLflow)
*Ports: Airflow UI (`8090`), MLflow UI (`5000`), MinIO Object Storage (`9000`, `9001`)*

While all that happens per transaction, an entirely separate massive ecosystem watches over the models:
- **MinIO Data Lake:** Spark periodically dumps Parquet backups of all processed pipelines into MinIO buckets (like `s3://fraud-data-lake/`).
- **Airflow DAGs:** Once a night, Apache Airflow activates (`fraud_detection_training_enhanced.py`). It orchestrates Spark clusters to read gigabytes of history from MinIO, apply SMOTE scaling, and retrain the XGBoost models on a cluster.
- **MLflow Tracking:** Airflow logs the exact F1/Precision metrics of the newly trained model to MLflow. If the new model beats the old one, Airflow writes the newly minted `model.json` to the `/opt/spark/models/` shared disk volume. Spark natively begins using the new file on its next micro-batch, updating its brain with zero downtime.
