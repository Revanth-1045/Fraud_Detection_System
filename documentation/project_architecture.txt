# Real-Time Fraud Detection Pipeline Architecture

This document provides a comprehensive, end-to-end breakdown of how this project operates. It is a production-grade, microservice-based machine learning pipeline designed to detect fraudulent financial transactions in real-time.

---

## 1. Step-by-Step Data Flow

### Step 1: Data Generation (The Producer)
- **Component**: `producer` container (`src/producer/main.py`)
- **How it starts**: This Python script acts as the origin of all data. It continuously generates synthetic financial transactions (including user IDs, amounts, GPS coordinates, and IP addresses). 
- **Fraud Injection**: It intentionally injects anomalous patterns into the stream. For example, it simulates "Impossible Travel" (a user teleporting across the globe in minutes), unusual transaction spikes, and known "fraud ring" behaviors.
- **Security**: The raw json payloads are encrypted using a symmetric cipher and then published to the Apache Kafka cluster on the `transactions` topic.

### Step 2: Ingestion & Decryption (The Gateway)
- **Components**: `kafka-1,2,3` cluster & `secure-gateway` container
- **How it works**: 
  - **Kafka** acts as a massive buffer, running 3 brokers for High Availability. It ensures that no transaction is lost even if downstream services momentarily lag.
  - The **Secure Gateway** aggressively consumes these encrypted messages from Kafka, decrypts them, and maps them to a fast `redis-ram-cache` to ensure data integrity and security before the data reaches the core processing engine.

### Step 3: Real-Time Processing & ML Inference (The Spark Engine)
- **Components**: `spark-inference` container running a PySpark script (`src/spark/streaming_job.py`), backed by a Spark Master and Worker nodes.
- **How it works**: This is the highly-parallelized "brain" of the operation. It reads the raw transaction stream from Kafka in micro-batches and executes several heavy operations:
  1. **Feature Engineering**: It transforms raw data into complex machine learning features. It calculates time mechanics (`hour_sin`, `hour_cos`), checks if it's the weekend, and computes velocity metrics (e.g., `tx_count_1h` - how many transactions this specific user made in the last hour).
  2. **Data Scaling**: It scales monetary values using a pre-fitted `RobustScaler` so the ML model can digest the numbers properly without being skewed by massive outliers.
  3. **XGBoost Inference**: It passes the 14 engineered features through a pre-trained XGBoost classification model (`/opt/spark/models/model.json`). The model returns a binary prediction.
  4. **Rule-Based Heuristics**: Alongside the ML model, it applies hardcoded heuristics (like checking if the user's IP belongs to a known ring, or if the physical distance between transactions is impossible).
  5. **Data Export**: Finally, it writes the processed results to a lightning-fast shared RAM disk (`dashboard_ram`). It exports the UI-friendly records to `recent.json` and the exact computed ML features to `live_features.json`.

### Step 4: The Dashboard API & SHAP Explainability
- **Component**: `web-app` container (`src/web_app/app.py`)
- **How it works**: A Flask backend that serves the beautiful frontend Web UI and provides the data endpoints.
  - **Live Feed**: The `/api/live` endpoint constantly monitors the `recent.json` file generated by Spark and serves the latest 50 transactions to the frontend table's heartbeat.
  - **SHAP Explanations**: When you click the **🧠 SHAP** button on a flagged transaction, the `/api/explain/<tx_id>` endpoint instantly searches the `live_features.json` file in reverse. Once it finds the transaction, it uses the Python `shap` library to deconstruct the XGBoost model's decision tree. It calculates *exactly* which features (like a high `terminal_fraud_rate` or an unusual `amount_z_score`) pushed the AI to flag it as fraud, and visualizes those weights on the screen.

### Step 5: Continuous Training (MLOps)
- **Components**: `airflow-scheduler`, `airflow-worker`, `mlflow-server`
- **How it works**: This pipeline is designed to learn over time. In the background, Apache Airflow acts as an orchestrator, running a scheduled Directed Acyclic Graph (DAG) for Model Retraining (`fraud_detection_training_enhanced.py`).
  - **Data Prep**: It gathers massive amounts of historical data and new feedback data.
  - **Class Imbalance**: It applies SMOTE (Synthetic Minority Over-sampling Technique) to generate synthetic fraud examples, assuring the model isn't biased toward "Clean" transactions simply because fraud is naturally rare.
  - **Training & Tracking**: It trains a new XGBoost ensemble model. For every training run, the accuracy, precision metrics, and hyperparameters are logged precisely into the **MLflow** server UI.
  - **Deployment**: If the new model performs better than the old one, it automatically overwrites the active `model.json` in the shared disk. The Spark streaming engine natively picks up this new file, effectively updating its brain without any system downtime.

## 2. Infrastructure Layer Breakdown

To support this massive data pipeline, several underlying infrastructure services are running concurrently:

- **ZooKeeper**: Manages the configuration, synchronization, and election processes for the Kafka brokers.
- **PostgreSQL**: A relational database acting as the primary backend for Airflow (task tracking) and MLflow (experiment tracking).
- **Redis (`redis-ram-cache`)**: Provides ultra-fast, in-memory caching. Used heavily by the Secure Gateway and the Web App for checking user statistics quickly without repeatedly querying heavy databases.
- **MinIO**: An S3-compatible object storage server. In enterprise setups, this acts as a data lake, ready to store massive datasets, historical logs, or massive model artifacts in a scalable way.
- **Prometheus & Grafana**: The monitoring tier. Prometheus actively scrapes health metrics and CPU/memory loads from all the containers (via exporters like `redis-exporter`), while Grafana visualizes these hardware and pipeline metrics in a separate dashboard (`localhost:3001`).
